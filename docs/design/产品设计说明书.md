# 产品设计说明书（AI 亲子教育 SOP 系统）

> 版本：0.2  
> 说明：本文档在 PRD《完整问题树 + H5 + 后台》基础上，结合当前阶段目标补充**非静态、后台驱动、AI 辅助生产、多端演进**等产品决策，随迭代更新版本号与修订记录。

## 1. 产品愿景

家长在孩子教育场景中遇到的具体问题时，能在移动端**快速定位问题 → 获得可执行的 SOP 与可复制提示词 → 真实改善亲子互动与学习结果**；运营侧可在后台持续维护分类与词条，并借助 AI **批量生成初稿、人工审核后入库发布**。

长期目标：用户持续使用线上系统，形成**用户资料、行为与反馈数据**的沉淀，支撑个性化建议与后续增值服务（含独立售卖可能）。

## 2. 目标用户与场景

- **家长（主用户）**：小学生至高中阶段孩子的家长，需在碎片时间用手机解决问题。
- **运营 / 教研 / 编辑（后台用户）**：维护分类树、词条、提示词包、上下架与审核；部分内容由 **AI 辅助生成** 后人工修订。
- **训练营学员**：课程与线上系统打包售卖；系统既是交付物，也是后续留存与转化的载体。

## 3. 核心体验（家长端）

1. **场景入口**：从「我现在遇到什么问题」进入；**1.0 以树状分类逐层下钻为主**（见第 4 节），辅助关键词 / 标签搜索；远期再强化「一句自然语言直接问」。
2. **叶子页统一结构**：问题本质、误区、原则、SOP、AI 提示词、示例、话术、任务、观察指标等（见 PRD 模板）。
3. **执行辅助**：一键复制提示词、阅读操作步骤；后续迭代可增加生成话术、打卡、收藏等（见 PRD 路线图）。

## 4. 产品形态演进：速查手册 → 语义问答

本节与 PRD 中的 V1 / V2 能力对应，把**商业上可交付的 1.0**和**远期能力**写清，避免工程上走偏（例如 1.0 不必上完整 RAG，但数据与 API 要预留扩展）。

### 4.1 1.0 / 近期：树状 FAQ +「AI 教育速查手册」

- **内容规模**：先以约 **100～200 个高质量叶子问题**为首批（可对应 PRD 的 MVP 清单再扩展），在后台按**多级分类树**分门别类、审核发布。
- **主要动线**：用户**不依赖**自然语言智能问答，而是像查手册一样 **一层层点进目录 → 打开具体问题的详情页**。
- **页内价值**：阅读 SOP、复制 **已预置、经审核的提示词**、按页内指导操作；定位是**可执行的速查与行动指南**，不是开放域聊天机器人。
- **搜索（可选增强）**：关键词 / 标签搜索作为补充，降低「只有点树才能找到」的摩擦；仍属 1.0 范围。

### 4.2 远期：自然语言提问 + 基于库的回答与生成

- **用户行为**：用户用**口语化、语义化**的问题提问，未必匹配目录文案，也未必愿意逐层点击。
- **系统行为**：**基于内容库**（已发布的词条、提示词、话术与结构化字段）做检索与综合，由 **AI 组织答案**；并可**动态生成或改写提示词**（仍建议保留「可复制」与合规边界）。
- **与 1.0 的关系**：1.0 沉淀的**结构化库**是远期的**唯一权威来源**；远期能力是在库之上的**检索 / 排序 / 生成层**，而不是另起一套未经审核的闲聊。

### 4.3 产品表述（对内对齐）

- **1.0**：「分层目录 + 固定词条 + 一键复制提示词的 AI 教育速查手册。」  
- **远期**：「以官方知识库为准的语义问答与个性化提示词输出（具体形态见架构文档中的能力分期）。」

## 5. 后台与内容生产（非纯手工）

系统必须支持**完整的内容发布闭环**，而不是静态页面手工改 HTML：

- **分类与词条管理**：多级分类、排序、可见性、年龄段、状态（草稿 / 待审 / 已发布）、版本。
- **AI 辅助生产（典型流程）**  
  - 输入：主题或父级问题（如「小学生作文」）、生成粒度（如拆成「开头」「结尾」子题）。  
  - **AI 生成**结构化初稿（各字段映射 PRD 中的内容模板）。  
  - **人工审核与修订**后**入库**。  
  - **发布**后家长端 H5（及后续小程序 / App）可检索与浏览。

人工审核环节用于把控教育质量与合规，避免未审核内容直接对用户可见。

## 6. 商业与打包形态

- **训练营独立售卖**：课程 + 本线上系统作为打包交付；账号与权益策略在 implementation 阶段细化（授权周期、是否绑定订单等）。
- **系统可能单独收费**：产品层保留「订阅 / 按年 / 与课程拆分售卖」等可能性；计费与支付接入在架构与后续迭代中预留扩展点。

## 7. 数据与反馈（留存目标）

前期即应考虑（不必第一期全部实现）：

- **用户侧**：基础画像、孩子学段、收藏、搜索词、浏览与复制行为（需隐私政策与用户同意）。
- **反馈**：有用 / 没用、不懂、希望补充的问题等（见 PRD 后台模块）。
- **目的**：优化内容排序与选题、支撑后续个性化与运营触达；与「孩子学习变好、家长能力提升」的长期叙事一致。**远期语义问答可反哺「搜不到的词」与选题迭代。**

## 8. 客户端演进路径

| 阶段 | 客户端 | 产品备注 |
| --- | --- | --- |
| 前期 | **H5** | 快速触达、嵌入训练营交付链路 |
| 中期 | **小程序**（微信等） | 便于分享、登录与消息触达；需与同一后端 API 对接 |
| 远期 | **原生 App**（可选） | 更强推送与端能力；仍应共享核心业务 API |

设计原则：**业务能力集中在服务端与统一 API**，前端多渠道薄封装，避免把业务规则写死在某一个 H5 页面里。
