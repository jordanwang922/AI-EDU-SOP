# 文档维护规范

## 适用范围

- `docs/development/开发日志.md`
- `docs/development/交接日志.md`

其他设计类、架构类文档不按「500 行清零」处理；若单文件过长，可采用分卷（如 `技术架构说明书-后端.md`）或章节拆分，由维护者判断。

## 开发日志（必填字段）

每次**开发会话结束**时，由执行开发的 AI 或人类开发者**追加一条**（从新到旧或从旧到新保持一致即可，本项目约定：**新记录追加在文件最上方**，紧跟说明头之后）。

每条至少包含：

1. **日期**（YYYY-MM-DD）
2. **执行者**：人类昵称或 AI 工具名（如 Cursor、Codex CLI、Antigravity）
3. **会话摘要**：本轮做了什么（可列要点）
4. **涉及路径**：主要改动的目录或文件（便于 Code Review）
5. **验证**：本地跑了哪些命令 / 待办（若本轮未验证可写「未跑」并说明原因）
6. **已知问题 / 风险提示**（无则写「无」）

## 交接日志（必填字段）

当存在**多人多 AI 接力**时，每一轮结束时追加一条，便于下一任快速上下文对齐。字段建议：

1. **日期**
2. **本轮执行者**（工具 + 可选模型标识）
3. **当前里程碑**：产品或工程阶段一句话
4. **已完成**：列表
5. **进行中 / 阻塞**：列表（阻塞需写依赖方）
6. **下一任建议第一步**：具体可执行的一条或几条
7. **环境提示**：分支名、需配置的密钥类型名称（勿写密钥值）、外部服务是否已接通

## 500 行归档规则

对 **开发日志** 与 **交接日志** 分别计数：

1. 当某个日志文件**总行数超过 500 行**（含空行与说明头）时，执行归档。
2. **归档操作**：
   - 将当前文件完整复制为：`docs/development/archive/<原名>-归档-YYYY-MM-DD.md`  
     若同日多次归档，可加后缀 `-序号`。
   - 将原文件**清空主体**，仅保留文件顶部的**固定说明头**（规则摘要 + 填写说明），从下一次记录开始重新追加。
3. 归档文件**只读不改**，历史追溯以归档文件为准。

说明头模板见各日志文件内嵌注释；修改说明头时需同步更新 `docs/README.md` 若涉及约定变更。

## 与 Git 的配合

- 日志追加与归档应与代码改动**同一提交或紧随其后**，避免「代码已合并但日志未更新」。
- Pull Request 描述中可简要引用当日交接日志条目编号或日期，方便审计。
