# 技术架构说明书（AI 亲子教育 SOP 系统）

> 版本：0.3  
> 说明：本文档给出**与产品目标一致**的技术原则与分层建议；具体框架选型可在仓库根目录 `README.md` 或后续 ADR 中落地。已与 `docs/design/产品设计说明书.md` v0.3 对齐（含 1.0 无登录、后台 AI 生成、数据预留）。

## 1. 架构目标

1. **不是纯静态站**：具备 **后台管理端 + API + 数据库（或等价持久化）**，所有面向用户的内容由后台发布。
2. **内容生产 AI 化**：支持「AI 生成 → 审核 → 入库 → 多端分发」；生成服务与业务 API 解耦，便于换模型与限流。
3. **多端演进**：同一套后端能力服务 **H5 → 小程序 → App**，避免重复建设。
4. **可运维、可审计**：GitHub 管理源码；部署与环境配置文档化；密钥不入库。

## 1.1 1.0 实施范围（工程视角）

| 区域 | 1.0 |
| --- | --- |
| **家长端 H5 API** | **无需登录**即可读分类树与已发布词条；不暴露「实时 AI 问答」端点。 |
| **管理后台** | **强鉴权**；必须包含 **「选题 → AI 生成完整词条模板 → 保存草稿 → 发布」** 的闭环 API 与 UI。 |
| **大模型** | 仅在**服务端**、仅对**后台**调用；家长端不携带模型 Key。 |
| **数据层** | 预留终端用户与订单相关表结构，详见 `数据模型与扩展性.md`。 |

## 2. 逻辑分层（推荐）

```
[ 家长端: H5 / 小程序 / App ]
           |
           v
      [ API 网关 / BFF（可选）]
           |
      [ 业务服务：鉴权、内容、搜索、反馈、埋点 ]
           |
      [ AI 内容服务：提示模板、生成任务、异步队列（可选）]
           |
      [ 数据层：关系库 + 搜索引擎（按阶段引入）]
```

- **BFF（Backend for Frontend）**：若各端差异大（登录体系、支付、摘要字段），可加薄 BFF；否则可由统一 REST/GraphQL 对外。
- **AI 内容服务**：负责调用大模型（如 **火山方舟 / 豆包** 等，可与既有项目共用密钥与 SDK 模式）、写入「草稿」状态内容、记录生成参数与版本；**发布动作**由后台人工触发（生成与审核可为同一操作者）。

## 3. 核心技术选型原则（待具体化）

| 领域 | 原则 |
| --- | --- |
| 前端（家长端） | H5 可用 Vue/React + 适配移动端的 UI；后续小程序使用各平台官方方案或跨端框架时，**尽量共用接口层与类型定义**。 |
| 前端（管理端） | 桌面 Web，权限路由、富文本或结构化表单编辑、预览、审核流。 |
| 后端 | 明确领域模型：分类、内容词条、提示词、版本、`ai_generation_jobs`、审核记录；**用户与订单表结构预留**（1.0 可不写入业务数据）。 |
| 搜索与问答 | **一期**：树状导航 + 词条 API；关键词 / 标签检索可用 PostgreSQL 全文检索或 `pg_trgm`。**远期**：向量检索 / RAG + LLM 组装回答与动态提示词，**仍以已发布库为 grounding**，与产品设计「语义问答」一致；体量上升后可加专用搜索引擎或向量库。 |
| 文件与素材 | 图片/附件走对象存储 CDN；URL 入库。 |
| AI | API Key 仅存服务端；生成任务可异步；记录 token 用量便于成本分析（可选）。 |
| 豆包 / 火山方舟 | 复用团队在其他项目中已接通的 **EndpointId + API Key**、语音等能力时，统一经 **AI 内容服务**封装，避免泄漏到家长端与静态资源。 |

具体语言栈（如 Node / Python / Go）与托管方式（云函数 vs 容器）在立项后写入 ADR。

### 3.1 后台「一键生成词条」接口（1.0 必备）

建议最小契约（示意）：

- **输入**：`topic` / `question_title`、可选 `age_group`、可选 `split_hints`（用于拆成子题）。  
- **输出**：结构化 JSON 映射 PRD 词条字段；服务端校验后写入 `contents` 为 `draft`。  
- **副作用**：写入 `ai_generation_jobs` 便于审计与成本统计。

流式输出（SSE）为可选增强，非阻塞首版交付。

## 4. 检索与问答能力分期（与产品路径对齐）

| 阶段 | 用户侧 | 技术侧重 |
| --- | --- | --- |
| **1.0** | 多层分类浏览；叶子详情页；可选关键词搜索 | 分类树 API、内容详情 API、复制次数统计；搜索走 DB 即可 |
| **增强** | 搜索体验优化、埋点与「无结果词」收集 | 索引优化、同义词表、运营后台看搜索日志 |
| **远期** | 自然语言提问；返回答案 + 可复制的生成/改写提示词 | 向量库或混合检索、引用来源、审核策略与护栏；可选异步生成 |

原则：**1.0 不要把「对话式大模型」做成唯一入口**；远期模块应复用同一套内容表与发布状态，避免两套真理源。

## 5. 内容模型与 PRD 字段对齐

PRD 已给出 `content`、`category`、`prompt` 等字段建议（见 `docs/prd/...V1.1.md` 第 3 节）。实现时建议：

- **slug** 唯一，便于 SEO 与分享链接（H5 与后续小程序路径统一用 ID 或 slug 策略需一次定好）。
- **状态机**：`draft` → `pending_review` → `published` → `archived`；AI 生成默认进入 `draft` 或 `pending_review`。
- **版本**：内容变更保留历史，满足运营回滚与审计。

## 6. 认证与账号（分期）

- **家长端 1.0**：**无登录态**；公开 `GET` 分类与已发布内容即可（仍需 HTTPS、可选 CDN、防盗链策略）。
- **家长端 后续**：手机号 / 微信 OAuth（小程序阶段重点）/ 邮箱等，按「训练营订单绑定」策略实现。
- **管理端**：强鉴权、角色（编辑、审核、管理员）、操作审计日志（建议）。

## 7. 从小程序与 App 演进的路径

1. **API 契约稳定**：OpenAPI 或合约测试，避免 H5 专属字段泄漏到协议层。  
2. **认证兼容**：预留 unionId / openId / JWT 等映射表，避免把某一端的登录写死。  
3. **推送与增长**：App 需单独配置；业务上仍以「内容库 + 用户数据」为中心。
