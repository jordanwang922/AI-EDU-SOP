# 数据模型与扩展性（1.0 无登录 / 后续登录与付费）

> 目标：**1.0 不上用户登录与付费**，但表结构一次性按「以后会需要」预留，避免重构迁移成本。

## 1. 原则

1. **用户与订单**：表可以**提前建好**，1.0 阶段无流量写入或仅后台种子数据；对外 API **不依赖**登录态。
2. **内容主链**：分类、词条、版本、发布状态为**核心域**，与是否有用户无关。
3. **外键策略**：用户相关字段 **NULLABLE**；匿名行为若有埋点可用 `session_id` / `device_fingerprint`（注意合规与隐私文案）。
4. **审计**：后台操作记 `operator_id`（编辑账号），与家长用户表分离。

## 2. 建议核心表（命名示意）

### 2.1 内容域

| 表 | 说明 |
| --- | --- |
| `categories` | 多级分类、`parent_id`、`sort_order`、`visibility` |
| `contents` | 叶子词条；结构化字段或 `JSONB` 存模板段落；`status`、`published_at` |
| `content_versions` | 可选：每次保存一条快照，便于回滚 |
| `prompt_templates` | 可选独立表或与 `contents` 内嵌；便于后台单独迭代提示词 |

状态建议：`draft` | `pending_review` | `published` | `archived`。

### 2.2 后台用户域（先有）

| 表 | 说明 |
| --- | --- |
| `admin_users` | 后台账号；角色 `editor` / `admin` 等 |
| `admin_audit_logs` | 谁在何时发布/下线/改写了哪条内容 |

### 2.3 AI 生成域（1.0 后台必备）

| 表 | 说明 |
| --- | --- |
| `ai_generation_jobs` | 一次「输入题目/主题 → 生成完整模板」的任务：`input`、`model`、`status`、`output_content_id`、`error`、`token_usage` |
| `ai_generation_events` | 可选：流式输出分段日志，便于排错 |

生成结果默认落到 `contents` 的 `draft`，由同一操作者在后台**审阅后发布**（产品约定：生成与审核可为同一人）。

### 2.4 家长用户域（预留，1.0 可空跑）

| 表 | 说明 |
| --- | --- |
| `end_users` | `id`、`phone`（可空）、`wechat_openid`（可空）、`created_at` … |
| `orders` / `entitlements` | 订单号、SKU、生效区间、与 `end_user_id` 绑定；1.0 不创建业务订单时可仅占位迁移 |

### 2.5 行为与反馈（可选，1.0）

| 表 | 说明 |
| --- | --- |
| `content_views` | 匿名亦可：`content_id`、`anonymous_session`、`created_at` |
| `copy_events` | 复制提示词计数：`content_id`、匿名会话 |
| `feedback` | 有用/没用；无登录时仍以匿名会话或可选填手机号 |

## 3. PostgreSQL 相关实践

- 广泛使用 **`JSONB`** 存模板字段，同时保留核心标量列（标题、slug、状态）便于索引。
- **`slug` + `UNIQUE`**；**`GIN`** 索引关键词数组或 `JSONB` 检索（按查询模式再定）。
- 将来向量检索可新增 **`embedding`** 列或独立向量扩展表，与 `contents.id` 一对一。
